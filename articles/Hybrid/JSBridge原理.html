# 👉 JSBridge 原理

# JSB 为了解决什么？

在移动端盛行的年代，技术选型上除了纯 H5 页面和纯 Native，还有一种常用的混合开发（Hybrid），混合开发是指使用多种开发模型开发 App，通常会涉及到两大类技术：原生 Native 和 Web H5。

混合开发按照渲染可分为下类：

  • 基于 Web 渲染的混合 App（Codova、NativeScript、微信浏览器、小程序）
  • 基于原生渲染的混合 App（ReactNative、Weex）

混合开发模式下，我们开发的 H5 页面运行在端上的 WebView 容器之中，很多业务场景下，H5 需要依赖端上提供的信息/能力，这时我们需要一个可以连接原生运行环境和 JS 运行环境的桥梁 。 这个桥梁就是 JSB （JSBridge），JSB 让 JS 侧和 Native 端得以实现双向通信。

所谓双向通信：

  • JS 向 Native 发送消息 : 调用相关功能、通知 Native 当前 JS 的相关状态等。
  • Native 向 JS 发送消息 : 回溯调用结果、消息推送、通知 JS 当前 Native 的状态等。

# 什么是WebView？

WebView 是移动端中的一个控件，也可以将其简单地理解成一个 App 内嵌的浏览器，它为 JS 运行提供了一个沙箱环境。WebView 能够加载并显示指定 URL 网页，拦截页面发出的请求、展示对话框等各种页面控制与交互功能，实现混合开发。JSB 的实现就依赖于 WebView 暴露的各种接口。

由于历史原因，IOS以 8 为分界，Android以 4.4 为分界，分为高低两个版本，不同版本不同内核。而它们的区别在于 —— 回调。高版本可以通过执行回调拿到 JS 执行完毕的返回值，然后准确进行下一步操作，而低版本无法执行回调。

# JSB 概念

JavaScript 是运行在一个单独的 JS Context 中（例如，WebView 的 Webkit 引擎、JSCore）。由于这些 Context 与原生运行环境的天然隔离，我们可以将这种情况与 RPC（Remote Procedure Call，远程过程调用）通信进行类比，将 Native 与 JavaScript 的每次互相调用看做一次 RPC 调用。如此一来我们可以按照通常的 RPC 方式来进行设计和实现。

在 JSBridge 的设计中，可以把前端看做 RPC 的客户端，把 Native 端看做 RPC 的服务器端（也可以把这个流程类比成 JSONP 的流程），从而实现 JSBridge 的主要逻辑就出现了：

  • 通信调用（Native 与 JS 双互通信）
  • 句柄解析调用。

# 实现原理

Native 和 Javascript 通信的原理是 JSBridge 实现的核心，实现方式可以各种各样，但是万变不离其宗。

# JavaScript -> Native 通信

# 注入式 API

注入 API 是一种比较自然的方式，主要原理是通过 WebView 提供的接口，APP 向 JS Context（window）中注入对象/方法，一般来说这个对象内的方法名与 Native 相关方法名是相同的，Web 端可以直接在全局 window 下使用这个暴露的全局JS对象，进而直接执行相应的 Native 代码逻辑，达到 JavaScript 调用 Native 的目的。

# 实现方式
# 安卓

可以通过addJavascriptInterface，向 webview window对象注入一个属性对象。这种方式传递参数类型要求是 android 和 js 两侧支持，只能是 int、string 这些基本类型（object需要stringify)。

但是此方法存在安全漏洞，在 Android4.2 以上提供 @javascriptInterface 注解来规避该漏洞(只能访问@javascriptInterface 注解方法)，但对于4.2以下版本则没有任何方法。(4.2 之前很多方案都采用拦截 prompt 的方式来实现。）

// Java: 定义一个类，提供一个接口，返回一个内容
class NativeApis {
  private Context ctx;
  NativeBridge(Context ctx) {
    this.ctx = ctx;
  }    
  
  // 增加JS调用接口
  @JavascriptInterface
  public String propose(String msg) { return "Yes!"; }
}

// 向webview注入一个全局对象（命名空间）
// 但在Android4.2以下存在漏洞，攻击者可以执行任意代码
// https://developer.android.com/reference/android/webkit/WebView#addJavascriptInterface(java.lang.Object,%20java.lang.String)
webView.addJavascriptInterface(new NativeApis(this), "Bridge");

// JS调用: 执行一个 native 的方法
result = window.Bridge.propose('say yes or no~')
# IOS
  • iOS 的 UIWebView 提供了 JavaScriptCore
JSContext *context = [uiWebView valueForKeyPath:@"documentView.webView.mainFrame.javaScriptContext"];

context[@"postBridgeMessage"] = ^(NSArray<NSArray *> *calls) {
    // Native 逻辑
};

// JS调用: 执行一个 native 的方法
window.postBridgeMessage('say yes or no~');
- iOS 的 WKWebView 提供了 WKScriptMessageHandler
// OC: 编写 Message 回调，并注册 Message Handler
@interface WKWebVIewVC ()<WKScriptMessageHandler>@implementation WKWebVIewVC- (void)viewDidLoad {
    ...
    // 注入对象，注册方法名，前端调用其方法时，Native 可以捕获到
    [_webView.configuration.userContentController addScriptMessageHandler:self name:@"nativeBridge（这是约定好的方法名）"];

    WKWebView wkWebView = [[WKWebView alloc] initWithFrame:self.view.frame configuration:configuration];

    // TODO 显示 WebView
}

// Javascript: 发消息给注入的 Message Handler
window.webkit.messageHandlers.Bridge.postMessage("say yes or no~");

# URL Schema 拦截

URL Schema 是类 URL 的一种请求格式，是为了方便 APP 互相调用设计的，格式如下：<protocol>://<host>/<path>?<qeury>#fragment。与我们网页 url 的主要区别是 protocol 和 host 都可以自定义，比如：jsbridge://showToast?text=hello。

Native 加载 WebView 之后，Web 侧发送的所有请求都会经过 WebView 组件，所以 Native 可以重写WebView 里的方法，从而拦截 Web 发起的请求，对请求的格式进行判断： 暂时无法在飞书文档外展示此内容

# 发送 URL 请求的方法
  • 方法：
    1. 带链接a标签，script等等 带链接标签多有默认行为，如a标签会打开新页面、script会阻塞页面加载等等。
    2. location.href 使用location.href会改变当前页地址，频繁切换 href 客户端拦截会丢失中间请求，也有可能引起页面跳转调用的丢失。
    3. 使用iframe.src 频繁改变 src 也有可能导致客户端拦截会丢失中间部分请求，但与location.href相比，iframe可以设置多个，因此有方案可以解决这一问题。
    4. 发送ajax请求 不少资料认为ajax在安卓上无法拦截，实际上ajax在android端上可以通过shouldInterceptRequest拦截get请求(post 等请求无法拿到请求体，所以只能使用get请求带上参数)，为何未使用 ajax 方案具体原因未知(有可能与ajax行为受浏览器影响较大有关).

综合来看使用iframe src成本较小，比较适合。针对频繁改变 src 有可能导致客户端拦截会丢失中间部分请求的问题，前端分开了触发端 jsb 时机和传递 jsb 调用信息的时机，维护了一个 jsb 调用队列，在触发 jsb 时，将 jsb 调用信息塞入队列；在传递 jsb 调用信息时，传递了当前时间调用队列里所有调用信息，传递完成后清空队列，回避了这个问题.

  • 使用 iframe.src 缺点：
    • 有 url 长度限制、数据格式限制的隐患。
    • 创建请求，需要一定的耗时，比注入 API 的方式调用同样的功能，耗时会较长。
    • 无法连续发送请求。 在同义词 JS Eventloop 内发送的请求，只有第一个能够成功传递到客户端。
# 处理过程
  1. H5 发送 jsbridge://dispatch_message 请求。
  2. 客户端拦截 H5 的请求，发现了 jsbridge:// 请求命中了 scheme 规则。
  3. 解析 schema 参数。
  4. 客户端执行完相关的操作，结束。 如果只是调起一个 schema，是没有办法知道执行的结果的。需要和 Native 约定一个全局 JS 函数，当有结果时，把结果传入回调函数。
# 实现方式

iframe.src的拦截方式主要有：

  • 安卓提供了 shouldOverrideUrlLoading 方法拦截
  • IOS UIWebView 使用 shouldStartLoadWithRequest，WKWebView 则使用 decidePolicyForNavigationAction
# Android

安卓实现拦截 webview 请求的方法有两种，通过 shouldOverrideUrlLoading 和 shouldInterceptRequest 都可以拦截 webview 请求并执行端能力返回。

// 解析要加载的 URL，如果是自定义 scheme，调用相关的函数
class MyWebViewClient extends WebViewClient {
    @Override
    public boolean shouldOverrideUrlLoading(WebView view, String url) {
        if (url.startsWith("jsbridge://")) {
          // 解析 // 后面的 action 和参数，调用相关的函数
        }
        return true;
    }
}

webView.setWebViewClient(new MyWebViewClient());
# IOS
  • UIWebView： shouldStartLoadWithRequest
(BOOL)webView:(UIWebView *)webView shouldStartLoadWithRequest:(NSURLRequest *)request navigationType:(UIWebViewNavigationType)navigationType
{
    //判断 URL 是否为 JSBridge 调用
    if ([request.URL isBridgeURL]) {
        [self handleCustomAction:URL];
        return NO;
    }
    
    return YES;
}
- WKWebView： decidePolicyForNavigationAction
 (void)webView:(WKWebView *)webView decidePolicyForNavigationAction:(WKNavigationAction *)navigationAction decisionHandler:(void (^)(WKNavigationActionPolicy))decisionHandler {
    // 判断 URL 是否为 JSBridge 调用
    if ([navigationAction.request.URL isBridgeURL]) {
        // Natvie 逻辑...
        decisionHandler(WKNavigationActionPolicyCancel);
        return;
    }
    else {
        decisionHandler(WKNavigationActionPolicyAllow);
    }
}

# 拦截 prompt 等原生 JS 方法

WebView 有一个方法，叫setWebChromeClient，可以设置WebChromeClient对象，而这个对象存在方法：onJsAlert、onJsConfirm、onJsPrompt、onConsoleMessage，当 JS 调用 window 对象的对应的方法，即window.alert、window.confirm、window.prompt、window.console，WebChromeClient 对象中的三个方法对应的就会被触发。

粗分的话，这个方法也算是 URL 拦截的一种，其主要的实现原理是： Native 劫持 Webview 的onJsAlert、onJsConfirm、onConsoleMessage、onJsPrompt方法并进行重写，即拦截 H5 发出的 alert、confirm、prompt、console.log。

以安卓为例，通过覆盖 WebChromeClient 类的 onJsPrompt 方法进行拦截：

// 重载 onJsPrompt 方法，提取 prompt 内容判断是否需要拦截
public class JSBridgeChromeClient extends WebChromeClient {
    @Overridepublic boolean onJsPrompt(WebView view, String url, String message, String defaultValue, JsPromptResult result) {
        result.confirm(JSBridge.call(view, message));
        return true;
    }
}

Android 4.2 以前，注入对象的接口是 addJavascriptInterface ，但由于安全原因不怎么使用，更多的会使用拦截 Prompt 的方式。这种方式没有太大缺点，也不存在连续发送时信息丢失。不过 iOS 的 UIWebView 不支持该方式（WKWebView 支持）。

# Native -> JavaScript 通信

JavaScript 作为解释性语言，最大的一个特性就是可以随时随地地通过解释器执行一段 JS 代码，所以可以将拼接的 JavaScript 代码字符串，传入 JS 解析器就可以执行，这里的 JS 解析器就是 WebView。

# 安卓

  • Android 4.4之前只能用 loadUrl 来实现，并且无法执行回调：
String jsCode = String.format("window.showWebDialog('%s')", text);

webView.loadUrl("javascript: " + jsCode);
  • Android 4.4之后提供了 evaluateJavascript 来执行 JS 代码，并且可以获取返回值执行回调：
String jsCode = String.format("(function() { return 'this'; })();");

webView.evaluateJavascript(jsCode, new ValueCallback<String>() {
   @Override
    public void onReceiveValue(String s) {
        Log.d("LogName", s); // Prints: "this"
    }
});

# IOS

  • UIWebView 使用 stringByEvaluatingJavaScriptFromString
// 执行当前加载页面中定义好的一个函数 propose()
result = [uiWebview stringByEvaluatingJavaScriptFromString:@"propose('Will you merry me?')"];
  • WKWebView使用 evaluateJavaScript
var config = new WKWebViewConfiguration();
var wkWebView = new WKWebView(UIScreen.MainScreen.Bounds, config);
var jsCode = (NSString) "document.getElementById('foo').innerHTML = 'bar'";

// 接收js传递的消息
WKJavascriptEvaluationResult handler = (NSObject result, NSError err) => { 
  if(err != null)
  {
    System.Console.WriteLine(err);
  }
  if(result != null)
  {
     System.Console.WriteLine(result);
  }
};

// 执行当前加载页面中定义好的一个函数 jsCode，并在打印调用结果
wkWebView.EvaluateJavaScript(jsCode, handler);

# 如何执行回调

通过上述介绍我们已经知道如何实现双端互相发送消息，但上述两个通信过程缺少了“回应”这一动作，接下来，我们就讲讲如何执行回调。

比如一个 JSB 调用是 Web 获取当前 App 信息，Native 拦截到 jsbridge://getAppInfo这样一个请求后将获取当前 App 信息，获取完成后如何让 Web 端拿到该信息呢？第一个简单的实现方法，类似于 JSONP 的实现：

  1. 随机生成一个callbackId，并挂载在全局 window 对象上，属性值就是回调函数，如：cb1234：getAppInfoCallBack；
  2. 创建 script 并发送请求，请求拼接上回调函数名称：src=jsbridge://getAppInfo?callback=jsonpCallback；
  3. 命中 schema 规则，Native 端拦截请求并解析参数
  4. Native 端执行相关逻辑操作后，直接执行全局对象上的对应回调函数 javascript:cb1234(appInfo)，这样一来整个通信过程就实现了闭环。
// Web
// 为防止事件名冲突，给每个 callback 设置一个唯一标识
const uniqueID = 1;

function webCallNative(event, params, callback) {
    if (typeof callback === 'Function') {
        const callbackID = 'jsb_cb_' + (uniqueID++) + '_' + Date.now();
        window[callbackID] = callback
    }
    const params = {callback: callbackID}
    // 构造 url schema
    const src = `jsbridge://${event}?${JSON.stringify(params)}`
    ...
}

webCallNative('getAppInfo', {}, (res) => { console.log(res) })

// Native
1. 解析传入的参数 'getAppInfo' 得知 Web 希望获取 AppInfo
2. 执行端逻辑获取 AppInfo
3. 执行参数中挂载在全局的 callback 方法，AppInfo 作为回调方法的参数
